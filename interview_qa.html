This is often the first question in an interview. Structure your answer to cover:

1. Your current role and responsibilities
2. Your relevant experience and achievements
3. What brings you to this position/company

Keep it concise (1-2 minutes) and relevant to the job you're applying for.

Object-oriented programming (OOP) is a programming paradigm based on the concept of "objects", which can contain data and code:

Key principles:
1. Encapsulation: Bundling data with methods that operate on that data
2. Abstraction: Hiding complex implementation details
3. Inheritance: Creating new classes from existing ones
4. Polymorphism: Ability to present the same interface for different forms

Example languages: Java, C++, Python, C#

Use the STAR method to structure your answer:

Situation: Briefly describe the context
Task: Explain what needed to be done
Action: Detail what you did to resolve it

SQL Databases:
- Relational databases with predefined schemas
- Use tables with rows and columns
- Excellent for complex queries and transactions
- Examples: MySQL, PostgreSQL, Oracle

NoSQL Databases:
- Non-relational, flexible schema
- Various data models (document, key-value, graph, etc.)
- Horizontally scalable, good for large datasets
- Examples: MongoDB, Cassandra, Redis

When to use: SQL for complex transactions, NoSQL for scalability and flexibility with unstructured data.

HTTP (HyperText Transfer Protocol):
- Unsecured protocol for transferring web pages
- Data is sent in plain text
- Uses port 80 by default
- Vulnerable to eavesdropping and man-in-the-middle attacks

HTTPS (HTTP Secure):
- Secure version of HTTP that encrypts data
- Uses SSL/TLS encryption
- Uses port 443 by default
- Provides authentication of the website
- Essential for securing sensitive data like logins, payments

Modern websites should always use HTTPS for security and SEO benefits.

REST (Representational State Transfer) is an architectural style for designing networked applications:

Key Principles:
1. Stateless: Each request contains all necessary information
2. Client-Server: Separation of concerns
3. Cacheable: Responses can be cached
4. Uniform Interface: Consistent resource identification and manipulation
5. Layered System: Components can't see beyond immediate layer

HTTP Methods:
- GET: Retrieve resource
- POST: Create resource
- PUT: Update resource
- DELETE: Remove resource

REST APIs typically return data in JSON or XML format.

Authentication:
- Verifying who a user is
- Confirms identity through credentials (username/password, biometrics, etc.)
- Answers the question: "Are you who you say you are?"
- Examples: Login systems, OAuth, multi-factor authentication

Authorization:
- Determining what a user can access
- Decides permissions after authentication
- Answers the question: "What are you allowed to do?"
- Examples: Role-based access control, permissions systems

In simple terms: Authentication is about identity, authorization is about permissions.

var:
- Function-scoped or globally-scoped
- Can be redeclared and updated
- Hoisted to top of scope
- Creates properties on the window object in browsers

let:
- Block-scoped (within {})
- Can be updated but not redeclared in same scope
- Not hoisted (technically hoisted but in TDZ)
- Preferred for variables that need reassignment

const:
- Block-scoped like let
- Cannot be updated or redeclared
- Must be initialized during declaration
- Preferred for constants and references that shouldn't change

Modern JS prefers let and const over var due to better scoping rules.

== (Loose Equality):
- Compares values after performing type coercion
- Converts types to match before comparison
- Can lead to unexpected results (e.g., 0 == false is true)
- Example: '5' == 5 evaluates to true

=== (Strict Equality):
- Compares both value and type
- No type conversion is performed
- More predictable and generally preferred
- Example: '5' === 5 evaluates to false

Similarly, != performs loose inequality comparison, while !== performs strict inequality comparison.

A closure is a function that has access to its own scope, the outer function's variables, and global variables, even after the outer function has returned:

Key Characteristics:
1. Functions have access to variables from their containing scope
2. The reference to outer variables persists even after outer function execution completes
3. Each closure has its own "memory" for the captured variables

Example:
function outer() {
  let count = 0;
  return function inner() {
    count++;
    return count;
  }
}
const counter = outer();
counter(); // returns 1
counter(); // returns 2

Closures are useful for data privacy, creating function factories, and in event handlers.

A Promise is an object representing the eventual completion (or failure) of an asynchronous operation:

States:
1. Pending: Initial state, neither fulfilled nor rejected
2. Fulfilled: Operation completed successfully
3. Rejected: Operation failed

Basic Syntax:
const promise = new Promise((resolve, reject) => {
  // Async operation
  if (success) resolve(value);
  else reject(error);
});

Consuming Promises:
promise.then(value => {
  // Handle success
}).catch(error => {
  // Handle error
});

Promises help avoid "callback hell" and provide better error handling for async operations.

GET:
- Used to request data from a specified resource
- Parameters are sent in the URL (visible in browser)
- Limited length (depends on browser, typically ~2000 chars)
- Should only retrieve data, not modify it
- Can be bookmarked and cached
- Idempotent (same request produces same result)

POST:
- Used to send data to a server to create/update a resource
- Parameters are sent in the request body (not visible in URL)
- No size limitations
- Can modify data on server
- Cannot be bookmarked
- Not idempotent (multiple identical requests may have different effects)

Generally use GET for safe operations, POST for operations that change state.

Synchronous Programming:
- Code executes line by line in sequence
- Each operation must complete before moving to next line
- Execution is blocked while waiting for operations to complete
- Simpler to understand and debug
- Can lead to poor performance with long-running operations

Asynchronous Programming:
- Operations can execute independently of the main program flow
- Program continues executing while waiting for operations to complete
- Uses callbacks, promises, or async/await to handle results
- More complex but better for performance
- Essential for I/O operations, network requests, etc.

Example: In JavaScript, setTimeout() is asynchronous, while most basic operations are synchronous.
